# 阅卷辅助器 · 第一阶段策划

> 版本 v0.1 · 基线文档
> 定位：单人使用、本地优先、自带工具进场的桌面阅卷工作流工具。

---

## 0. 一句话

把奥赛纸质大题的判卷里**所有非判断性的损耗**（翻页寻址、手眼往返录入、事后聚合）全部消灭，只留下"看题—给分"这一个纯判断动作，并让这个动作在键盘上快到极致。

这不是一个扫描工具，也不是一个智能判分平台。它的全部价值锚点是**判分循环**。

---

## 1. 范围与边界（最重要的一节）

第一阶段唯一目标：**一个人，对着一批已经扫好的干净图，把一场奥赛（<8 道大题）批到"极爽"，并一键出成绩表。**

### 1.1 第一阶段做（In scope）

- 一场考试 = 一个自包含文件夹（图 + 一个 SQLite 库），可整体拎走归档。
- 从一个图片文件夹导入（Ingest）。
- 靠采集顺序 + 手动兜底给每张图打上 `(学生, 题号)` 标签（Label）。
- **基于评分细则（rubric）的横向批题键盘循环（Grade）——命门，全部精力砸这里。**
- 导出 CSV / PDF 成绩表，含按题得分率（Export）。

### 1.2 第一阶段明确不做（Explicit non-goals）

以下每一项都能单独吞掉一个月，且服务的是"我现在并不拥有的市场"或"我一个人用不到的场景"。全部推迟：

- **成像/矫正模块**：第一阶段直接外包给扫描全能王 / 夸克扫描王免费版，本工具从"一批干净图"这个起点接手。不写 OpenCV/ONNX 矫正链。
- **AI 辅助判分**：不做。相关约束见 §7。
- **多用户 / 局域网协作**：不做。架构留门（§3），但不实现。
- **二维码 / 条码 / OCR 自动识别学生与题号**：不做，靠物理顺序 + 手动。
- **页内题目切分 / 版面分析 / 框线检测 / OMR 客观题**：奥赛全大题、一面一题，从物理上就不存在这些问题，永远不做。
- **dewarp / 弯曲页面矫正 / 深度学习成像**：杀鸡用牛刀，不做。

> 判据：动手前对每个想加的功能问一遍——"这是**我一个人批这场奥赛需要的**，还是**我想做的**？"后者一律推迟。

## 2. 技术选型

**Tauri v2 + Rust/Axum(备用) + SQLite + Vue3。** 直接复用 boothkernel 架构，零迁移成本。

| 维度 | 选择 | 理由 |
|---|---|---|
| 形态 | 桌面 App（Tauri），**不是网站** | 本地优先、断网可用、自带工具进场；数据不出本地 = 责任边界干净 |
| 后端核心 | Rust crate `grading-core` | 逻辑与外壳解耦，将来要多机就再包一层 Axum，不重写 |
| 存储 | SQLite（每场一库） | 单人单写者，量级几百条记录，天然无并发问题 |
| 前端 | Vue3 | 复用现有栈；判分界面靠键盘，不靠花哨组件 |
| 视觉 | monolith 设计系统 | 冷、减法、高信息密度、几何。判分工具要的就是这个 |

> 不选 web 的唯一例外场景是"多助教局域网并行批同一场"——那是被明确推迟的 non-goal（§1.2）。默认单机。

---

## 3. 架构：五段流水线（第一阶段实现其中四段）

核心逻辑抽成独立 crate `grading-core`，现在用 Tauri IPC 暴露给前端；将来真要多机，同一个 core 外面再包一层 Axum 对外服务即可（同 THTK-Studio-2 的"共享 core 暴露多接口"模式）。

```
① Ingest   导入      —— 读图片文件夹（第一阶段：最简，最后做）
② Preprocess 成像    —— 【第一阶段跳过，外包给扫描全能王】
③ Label    打标签    —— (学生, 题号)：物理顺序 + 手动兜底
④ Grade    判分      —— 横向批题键盘循环【命门，最先做】
⑤ Export   导出      —— CSV / PDF 成绩表 + 按题得分率
```

采集与成像、判分彻底解耦。第一阶段的输入契约是：**一批扫描全能王导出的干净图（单页出图，见 §6）。**

## 4. 数据模型

关键决策：**评分不存扁平总分，存分步给分的 rubric 项。** 奥赛判卷本就是分步给分（建系 4 分、列方程 5 分、积分 6 分……）。把它建进模型，一箭三雕：判分时数字键 toggle 每步（快）、同一 rubric 应用到全部答卷（标准稳）、每步得失全留痕（可审计 + 教研金矿）。

```
exam         (id, name, date)
problem      (id, exam_id, number, max_score, title)
rubric_item  (id, problem_id, order, label, points)   -- 评分细则每一步
student      (id, exam_id, name/外部号, seat)
page         (id, exam_id, student_id, problem_number,
              image_path, status)                     -- 一张图一条
score        (id, student_id, problem_id, total,
              grader, submitted_at)
score_item   (id, score_id, rubric_item_id, earned)   -- 这一步给了几分
```

两个要点：

1. **判分单元是 `(学生, 题号)`，不是 page。** 奥赛一道大题可能写到第二面溢出，一个 `(学生, 题号)` 可挂多张 page，判分视图把它们一起滚动显示。一面一题是常态，但溢出情形必须在模型里留好，否则遇到就崩。
2. **`score_item` 是相比"只存总分"最大的增值。** 批完自动能出"83% 的人栽在积分那步"这种题目级诊断，几乎零额外成本，一定要做。

存储组织：`/某场考试/{images/, exam.db}`，整个目录可拎走、归档、交接。

## 5. 命门：判分循环 UX

这是整个软件**唯一每天点几百上千次**的地方，90% 的好用度在这里。

> **动手写代码前，先拿纸把按键序列和界面框图画死，再写。** 这部分没有技术含量，但极吃细节。

### 5.1 模式：横向批题（题内纵切）

选定一道题 → 系统把该题所有 `(学生, 题号)` 单元排队 → 逐个呈现。连着批同一题几十上百份，判分标准最稳、速度最快。这是学校成熟系统跑了二十年的"按题切割分发"的单机版。

### 5.2 单屏布局

```
┌─────────────────────────────┬───────────────┐
│                             │  第 3 题 (20分) │
│      答卷图像               │               │
│   （滚轮缩放，看清密集手写）  │  1 建系受力 4  │
│                             │  2 列方程   5  │
│   溢出的第二面在下方续接      │  3 积分     6  │
│                             │  4 边界     3  │
│                             │  5 结果     2  │
│                             │  ─────────    │
│                             │  当前：14 / 20 │
├─────────────────────────────┴───────────────┤
│  本题进度 34 / 50        [F]存疑  [J]下一存疑  │
└──────────────────────────────────────────────┘
```

### 5.3 按键序列（全键盘、盲操作、鼠标碰都不用碰）

| 键 | 动作 |
|---|---|
| `1`–`9` | toggle 对应 rubric 项得/不得；需步内部分分则该项再按进副输入 |
| `Enter` / `空格` | 提交并下一份 |
| `←` / `Backspace` | 回上一份改分 |
| `F` | 标记存疑待复看 |
| `J` | 跳到下一个存疑 |
| 滚轮 / `+` `-` | 图像缩放 |

总分随 toggle 实时求和显示在角上。进度条常驻。

## 6. 采集方案（第一阶段：外包）

**用扫描全能王或夸克扫描王免费版，单页/单学生出图，绕开多页 PDF 的页数限制，忍受角落水印（不碰作答区，不影响判分）。**

- 页数限制针对"单个多页 PDF"，你的采集单元是一张张独立图 —— **不使用多页 PDF 功能，限制就不成立。**
- **不下破解版 APK。** 拿来路不明的破解版处理学生答卷，数据安全与法律上都是给自己埋雷，与"把自己摘干净"的原则相悖。
- 推荐路子：一叠一叠无脑扫、导出单页图，**不在 App 端费心命名/分组**，归类全部留给桌面系统（§7 的物理顺序方案）。手机是最烂的做数据组织的地方，它只负责把图弄干净。

> 中期演进（非第一阶段）：把成像段收进本工具 —— **jscanify（前端 OpenCV.js，Vue 直接调）** 或 OpenCV 经典 pipeline（边缘→四角→透视矫正→自适应阈值）。届时无水印、无限制、数据不出本地，彻底摆脱第三方 App。量再大就上带 ADF 的文档扫描仪。

## 7. 打标签方案（Label）

给每张图补 `(学生, 题号)`。第一阶段只用两档，够了：

1. **物理有序采集（主）**：每学生卷子按固定页序叠好，一叠一叠过。软件靠"每 N 张一个学生、第 k 张 = 第 k 题"纯顺序补全标签，零识别代码。配 **"每叠固定张数"校验**：扫完一叠数张数，不够就报警——防漏扫/多扫导致的错位。
2. **手动兜底（必备）**：识别不了或学生答串了的进"待确认"队列。因为切分粒度是**整页不是页内区域**，手动改标签是**零成本**操作——瞄一眼点一下改题号即可，不涉及任何像素操作。这正是"一面一题"最大的红利。

**必须有的界面**：一个"这批图 ↔ (学生,题号) 对应关系"的可视化确认页，导入后扫一眼、拖一拖修正错位。

> 二维码/条码自动识别是中期优化（比 OCR 手写数字可靠一个数量级），但第一阶段不做。**永远不依赖学生手写的页码/题号**，漏写错写会坑你，顶多做辅助校验。

---

## 8. AI 层：第一阶段不做；将来做也守死一条线

**AI 当分诊台，不当法官。最终分永远是你按的键。**

奥赛物理大题尤其要克制：模型判推理给分点不可靠到会坑你，教培场景传出"AI 打的分"是纯负债。将来（非第一阶段）若加，只允许做两件安全的事：

- **组织**：把解法相近的答卷聚在一起让你连着批（视觉相似 ≠ 解法相似，只当粗排序）。
- **质检**：批完挑出"很像但给分差很多"的对子、疑似空白/跑题的，提醒你复看。

**架构硬约束（写死在 core 里）**：AI 模块只能写建议字段和排序，**永远不能写 `score.total`**。

## 9. 构建顺序与里程碑

先验证最核心的爽感，再补齐管道。

| 里程碑 | 内容 | 验收 |
|---|---|---|
| **M1 判分循环** | 用假数据（一个装几十张图的文件夹 + 手填 rubric），把 §5 的键盘循环做到"一个人批起来极爽" | 能全键盘横向批完一道题、实时总分、回改、存疑 |
| **M2 打标签** | §7 物理顺序 + 手动兜底 + 对应关系确认界面 | 导入乱序图能正确切成 (学生,题号)，错位能一眼修正 |
| **M3 导出** | CSV / PDF 成绩表 + 按题得分率（吃 `score_item`） | 一键出总分排名表 + 各题得分率 |
| **M4 导入** | 读图片文件夹（最简单，最后补） | 指向扫描全能王导出目录即可开批 |

> M1 先行的理由：判分循环是价值锚点，也最能暴露 UX 问题。管道其余部分都是为它服务的。

## 10. 风险与对策

| 风险 | 对策 |
|---|---|
| **过度设计**（最大风险）：忍不住去做通用版面分析、AI 判分、多用户 | §1.2 non-goal 清单是硬边界；每个新功能问"需要 vs 想做" |
| **顺序流错位**：某叠漏扫/多扫一张，后面全错 | "每叠固定张数"校验报警 + 手动确认界面兜底 |
| **溢出页遗漏**：奥赛大题写到第二面 | 数据模型 `(学生,题号)` 挂多 page，判分视图滚动显示 |
| **并发妖怪的应激反应**：被东方炒炒币的 PG 写瓶颈吓到，过度设计并发 | 本工具单机单用户本地 SQLite，单写者天然成立，怎么简单怎么来 |
| **交付物带第三方水印**：将来把批注卷/成绩单交还机构，观感减分 | 到那步再把成像段换成自有 jscanify/OpenCV 链或 ADF 扫描仪 |

## 11. 第一阶段完成的定义（Definition of Done）

指向一个装着扫描全能王导出图的文件夹，能够：

1. 导入并（靠顺序 + 手动）正确标好每张图的 `(学生, 题号)`；
2. 全键盘、横向批题、基于 rubric 分步给分，一个人批得极爽；
3. 一键导出总分排名表 + 各题得分率；
4. 整场数据是一个可拎走的自包含文件夹。

达成即第一阶段结束。二维码识别、AI 分诊、自有成像链、多用户，都是**验证了第一阶段爽感之后**才谈的下一阶段。

---

*把不确定性挡在你能控制的物理边界之外，而不是在下游用算法去救——这是贯穿整套设计的唯一原则。*
