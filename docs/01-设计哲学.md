# 01 · 设计哲学

本文提炼自设计定稿 `superpowers/specs/2026-07-02-阅卷辅助器-design.md`（v0.2，在 v0.1 基线上经一轮 grilling 修订）。它回答的是「**为什么这么设计**」——每一处决策都附理由，并尽量记下被否决的替代方案。

## 0. 唯一价值锚：判分循环

> 把奥赛纸质大题判卷里**所有非判断性的损耗**（翻页寻址、手眼往返录入、事后聚合）全部消灭，只留下「看题—给分」这一个纯判断动作，并让它在键盘上快到极致。价值锚点是**判分循环**，仅此。（§0）

一切都为这个循环服务。流水线的其余四段（设置 / 摄入 / 标注 / 导出）存在的意义，只是让这个循环**能开始、能收尾**。

- **命门意识**：判分 UX 是「唯一每天点几百上千次的地方，90% 的好用度在这里」。所以它**最先做**（M1），也最先暴露问题（§5、§11）。
- 这条原则的工程投影：判分逻辑写成**无副作用的纯 reducer**（`useGradeKeys.ts`），键位序列被单元测试钉死；UI 追求单屏、无鼠标往返。详见 [架构与模块 · 纯键盘 reducer](02-架构与模块.md#纯键盘-reducer-模式)。

## 1. 本地优先、单人单用户、自包含

- **定位**：单人使用、本地优先、自带工具进场的桌面阅卷工作流工具。技术上「默认单机、本地 SQLite、单写者，无并发问题」（§2）。数据不出本地，无网络。
- **克制过度设计**：风险表把「并发妖怪的应激反应」列为要压住的冲动——「单机单用户本地 SQLite，单写者天然成立，怎么简单怎么来」（§12）。
- **可移植 / 自包含**：一场考试 = 一个可拎走的目录 `{exam.db, images/}`，整体可归档、交接。导入时把图**拷进** `images/`，原件不动（§4.3）。`page.image_path` **只存文件名**，显示时解析为 `<考试目录>/images/<文件名>`，保证目录可整体搬迁。

## 2. 关键设计决策（含被否决方案）

### 2.1 部分分只存题目级总分，砍掉题内步级录入 —— *grilling 修订*
判分单元 `(学生, 题号)` 只存**一个总分数字** `score.total`，不录题内每步得分。
- **否决的 v0.1 方案**：分步存 `score_item`（每个 rubric 步给几分）。
- **为什么**：奥赛判卷细化到「每道题」的诊断粒度已足够；再细到题内步骤，录入负担太重，且部分步骤本质是连续给分、拆不动。题目级得分率成立，题内步级诊断推迟。

### 2.2 `score_item` / `rubric_step` 建表占位、v1 不录不依赖
两张表建了但空着，「为将来的题内步级诊断留一扇向前兼容的门」（§4.1）。当前**绝不承载任何逻辑**。

### 2.3 `score.total` 唯一写入路径
`score.total` **只由 `set_score` 写**——评语、导入、将来的「AI」都不写它。
- **AI 硬约束**（§10）：AI 只当「分诊台」，不当「法官」，最终分永远是你按的键。写死在 core 里：AI 只能写建议字段与排序，**永不写 `score.total`**。
- 评语走独立的 `set_comment`：可以先写评语不给分（建一条 `Ungraded` 行只带评语），给分时也不冲掉评语，且从不碰 `total`。

### 2.4 快捷档位 `score_preset` 绑数字键 1–9，顺带白捡粗诊断
判分不逐步 toggle，靠**每题预置的绝对分值档位**一键落总分，配手动兜底（§5.2、§8）。
- **否决的方案**：相对表达式（「满分−n」）、跨题模板——全砍。「前两问=9」对软件就是数字 9。
- **白捡的诊断**：`score` 存 `preset_id`，一句 `WHERE preset_id=该档` 就能捞出「卡在建系那步的人」。因此指引是：**按「卡在哪一步」设计档位；能用档位就别手动**——手动输入 `preset_id=NULL`，这个信号就丢了。
- 建题时自动预置三档：`满分`(键 7,=max) / `零分`(键 8) / `空白`(键 9)。

### 2.5 花名册放在「考试设置」层 —— *grilling 新增段*
题目、满分、档位、**花名册**同属考试骨架，全在设置层录好（§8）。
- **为什么**：花名册既是标注段绑身份的**选人来源**，也是**到场清单**（缺考检测）——一处录入，两处复用。

### 2.6 三态 + 学生层缺考
判分单元三态：`未判 Ungraded` / `存疑 Flagged` / `已判 Graded`；学生层另有 `缺考 Absent`（花名册有、无任何 page 绑定），与「已交但未判」严格区分（§9.1）。
- **为什么**：物理上区分「没答=真 0 分」（空白页进队列被打 0）与「没批=未判」。缺考不是存储态，而是导出时**派生**出来的。

### 2.7 导出永不静默填 0
未判 / 存疑**绝不静默填 0**：表里留空或标 `?`，缺考整行标「缺考」（§9.1）。
- **为什么**：否则一个被跳过没批的学生，会在成绩表里变成「这题真得 0 分」的惨案，且事后查不出。
- **软提示不硬拦**（单人工具，你为结果负责）：导出前弹覆盖率 + 到场确认（如「花名册 50 人，缺考 1；已判 291、存疑 2、未判 1，仍要导出吗？」）。

### 2.8 排名仅当「该生所有单元均已判」才输出
只要某生还有未判/存疑，其总分未定，排名对他无意义。**未达全部已判前不出排名列**；全员（非缺考）都判完才计算，用标准竞赛名次（并列同名次、后跳位，如 1,1,3）（§9.1）。
- **总分语义**：存疑的临时分**计入**临时总分（是真敲的数，带存疑标）；未判不贡献分（留空，不当 0）。所以有洞时总分是**下界/临时值**。绝不用带洞数据排出一个「像真的」名次。

### 2.9 `problem_number = 0` = 姓名页/封面
姓名页不判分、速览可见、用于身份复核；在判分队列、答题页计数里处处排除（§4.2、§7.1）。
- **为什么**：姓名页让「学生边界」从「数张数猜」变成「肉眼可见」，是标注段能简化的支点。

### 2.10 §7.0 承重不变量：叠内答题页数 == 题数 N —— *grilling 显式声明*
叠内自动编题号成立的**唯一前提**是「每个学生的答题页数 == 题数 N」。
- **识别出的静默错位风险**：奥赛放弃某题是常态。若一叠是 `[姓名页, Q1, Q2, Q4, Q5]`（跳了 Q3 又没占位页），纯自增会把 Q4 静默标成「题3」、其后全体后移，且事后查不出。
- **物理 + 软件双层守护**：物理层「缺答塞一张空白页占位」（每生严格 = 姓名页 + N 张）；软件层**绝不盲目自增**——`==N` 才走自增快路径，`≠N` 立即报警、强制逐页指认。正确性押在软件层（对不上就喊），不押在「人一定记得塞空白页」这条纪律上。

### 2.11 交卷乱序 → 姓名页定边界 + 从花名册选人 —— *grilling 修订*
- **否决的 v0.1 假设**：「扫描序 = 名单序」。修正为：交卷是人肉乱序的，纯顺序锚不住「这叠是谁」。
- **拆开两个被 v0.1 捆在一起的假设**：①「这叠是谁」（跨学生身份）——乱序，靠姓名页 + 花名册绑定；②「叠内这页是第几题」——只要学生内不散，`第k页=第k题` 仍成立。
- **身份绑定走「从花名册选人」**（搜姓名/键考号），不手填考号——花名册是身份的单一真源，比对着卷面手敲更顺、更不易错（§7.2）。
- **逃生口**：查无此人 → 允许临时新增学生 或 标「待匹配」（确认总表/导出前单列出来，逼你收尾）；标注结束后无 page 绑定者自动列为缺考（§7.2–§7.3）。

### 2.12 `Enter` 上下文相关，绝不一键两义
手动输入框内 `Enter` = 确认数字并**留在当前单元**（给一眼复核机会）；爽批态 `Enter` = 提交并跳下一份。所以手动给分要按两次 `Enter`（§5.2）。这是刻意为之，避免一键两义。

### 2.13 两条导航轴分开，轴 2 纯只读
轴 1（队列前进/后退、`G` 随机跳转、`J` 跳存疑）与轴 2（速览别的答题纸）独立绑键。轴 2 **纯只读**：右侧评分面板始终锚定当前 `(学生,题号)`，绝不会误落到正瞄着的那页上（§5.3）。
- 判分时若发现标注错了，**当场不改**，回标注阶段修——不在只读判分界面里塞标签编辑，避免两个阶段职责混淆。
- `G` 队列总览随机跳转，解决「批到第 40 份想起第 5 份判错、退 35 次不可接受」。

### 2.14 采集外包，成像不自建（第一阶段）
采集外包给扫描全能王/夸克扫描王免费版：手机只负责把图弄干净，不做命名/分组（§6）。中期再把成像段（jscanify/OpenCV）收进来。

### 2.15 图片走 IPC 字节传输 + PDF 页渲染成 JPEG
- 读图不走 Tauri asset 协议，而是命令返回字节——一次一张几百 KB~2MB 够用，且**彻底绕开 asset 协议的动态作用域授权坑**（考试目录是动态的，capability 极易漏——见 M3 补 `dialog:allow-save` 的教训）。文件名有安全护栏（拒 `/ \ .. :`）。
- PDF 页渲染成 **JPEG**（不是 PNG）再跨 IPC：扫描页 PNG 多 MB，`number[]` 传输是瓶颈。
- 报表 PDF 不引 Rust PDF 库，走「可打印 HTML + `window.print()` 另存」，绕开 Rust 中文字体嵌入坑。

## 3. 贯穿全篇的唯一原则

> 把不确定性挡在你能控制的**物理边界**之外（姓名页定边界、花名册当到场清单、三态可见），而不是在下游用算法去救。——这是贯穿整套设计的唯一原则。（design.md 文末）

「§2.8 排名门槛」「§2.7 不填 0」「§2.10 页数==N 报警」都是它的具体投影：宁可提前暴露、强制人来指认，也不让脏数据静默流到成绩表里。

## 4. 非目标 / YAGNI / 推迟项

**判据**（反过度设计，§1、§12）：动手前问一句——「这是**我一个人批这场奥赛需要的**，还是**我想做的**？」

第一阶段明确**不做**（§1）：成像/矫正、AI 判分、多用户/协作、二维码/条码/OCR 自动识别、页内切分/版面分析/OMR、dewarp。

具体推迟：
- **题内步级诊断**（`score_item`/`rubric_step`）：建表占位不录，验证爽感后再谈（§4.1、§13）。
- **OCR 姓名/手写考号**：v1 不做（手写识别不可靠），身份一律人工从花名册选；将来做也只当辅助校验（§7.4）。
- **二维码/条码**：中期优化；永不依赖学生手写的页码/题号（§7.4）。
- **自有成像链**：第一阶段外包，中期再收（§6）。
- **AI 层**：第一阶段不做；将来只允许两件安全事——「组织」（把相似答卷聚在一起连批）与「质检」（挑很像但给分差很多的对子提醒复看），**永不写 `score.total`**（§10）。

> DoD（§13）：题内步级诊断、OCR/二维码、AI 分诊、自有成像链、多用户，都是**验证了第一阶段爽感之后**才谈的下一阶段。
